每一门的编程语言都存在类型转换，即将一个值从一种类型转换为另一种类型，JS也不例外。但是JS的类型转换与其他语言不同，一般JS将类型转换统称为强制类型转换，我们可以将它分为隐式强制类型转换与显式类型转换。JS的强制类型转换只会返回简单值（标量基本类型值，number/string/boolean/symbol/null/undefined），不会返回对象和函数。

显式与隐式是相对的，或许上面的a + ""对你来说你，你很清楚它发生了什么，它相对你来说就是“显式”的。

抽象操作

在更深入了解JS的强转之前，我们先来一块垫脚石，为后续打好基础。所谓的抽象操作，就是仅内部使用的操作，我们开发者是无法调用，所以我们需要知道有这一回事。JS的抽象操作有以下几种：

  • ToPrimitive
  • ToString
  • ToNumber
  • ToBoolean

ToPrimitive

ToPrimitive操作将输入的参数强制类型转换为非对象类型(non-Object type)，可以理解为转换成标量基本类型值。To(变成)Primitive(原始的)=>变成原始的，原始可以看成是简单值（标量基本类型值），这样就容易记忆。

  • 如果输入的参数是非对象类型，将返回自身，即不进行转换
  • 如果输入的参数是对象类型，将进行调用对象内部方法[[DefaultValue]]，返回经过转换后的非对象类型，即简单值

[[DefaultValue]]

该方法可以传入一个参数hint，[[DefaultValue]](hint)

  • 如果参数hint是number类型，那么将采用以下步骤

    1. 检查该值是否存在valueOf()方法
    2. 如果存在valueOf()方法
      • 调用valueOf()方法，返回结果为val。
      • 如果val是基础类型值，那么就返回val(方法就结束了)。否则进行下一个步骤
    3. 检查该值是否存在toString()方法
    4. 如果存在toString()方法
      • 调用toString()方法，返回结果为str。
      • 如果str是基础类型值，那么就返回str(方法就结束了)。否则进行下一个步骤
    5. 抛出TypeError错误

  • 如果参数hint是string类型，那么与第一种情况区别不大，主要是先检查toString()的存在，再检查valueOf()的存在

  • 如果没有传入参数hint，那么方法的步骤将与参数hint是number类型的情况一样

ToString

ToString操作是将输入的参数强制类型转换为字符串类型

  • null -> "null"
  • undefined -> "undefined"
  • true/false -> "true"/"false"
  • 数字 -> 遵循通用规则转换为相应的字符串
  • 对象(Object type) -> 调用对象的toString()方法
    • 除非对象对toString()进行自定义，否则，一般toString()（Object.prototype.toString()）返回内部属性[[Class]]的值，比如"[object Object]"。Ps:一般我们就是用这个方法判断对象类型

ToNumber

ToNumber操作是将输入的参数强制类型转换为数字类型

  • null -> 0
  • undefined -> NaN
  • true/false -> 1/0
  • 对象(Object type) -> 进行ToPrimitive操作，hint参数为number类型
  • 字符串 -> 基本遵循数字常量的相关规则。如果处理失败将返回NaN(Not a Number)，下面是需要注意的几个情况
    • [] -> 0 (数组体为空)
    • "" -> 0(字符串内容为空或空白符，如空格、回车符\n,\r,\n\r、制表符\t等)

ToBoolean

ToBoolean操作是将输入的参数强制类型转换为布尔类型

对于ToBoolean来说，只要是假值的强制类型转换结果都是false，假值列表以外的强换结果都是为true

假值(false value)列表:

  • null
  • undefined
  • false
  • +0、-0、NaN
  • ""  (空字符串，即没有任何内容，也不包括空白符，别混淆ToNumber的情况)

"false"/"''"/"0"/[]/{}等等这些都是假值列表以外，所以强制类型转换结果都是true

解析数字字符串

parseInt(str)可以将传入的数字字符串解析为数字。嗯，是解析不是转换。

通过上面的代码，我们可以理解到什么是解析了，也可以知道parseInt()会从左到右解析字符串，直到遇到非数字字符(非有效字符)为止。如果第一个字符不能被转换成数字，parseInt()返回NaN。

parseInt()可以传入非字符串类型的值，但是不推荐。因为，向parseInt()传入非字符串类型参数，那么将会首先将该参数强制转换为字符串（即使是数字类型，同样会被处理），注意这是隐式的，一不小心就出乎我们的意料了。

其次parseInt()有个坑，parseInt(str, radix)有两个参数，第二个参数（可选）指定转换的基数(进制)。在ES5以前，如果没有指定第二个参数，那么parseInt()将根据字符串的第一个字符来决定转换的基数。比如第一个字符为x/X的字符串会转换为十六进制，如果是0，则转换为八进制。

ES5开始在没有指定基数，或者基数为 0 的情况下，JavaScript 作如下处理：

  • 如果字符串 string 以”0x”或者”0X”开头, 则基数是16 (16进制).
  • 如果字符串 string 以”0”开头, 基数是8（八进制）或者10（十进制），那么具体是哪个基数由实现环境决定。ES5规定使用十进制，但是并不是所有的浏览器都遵循这个规定。因此，永远都要明确给出radix参数的值。
  • 如果字符串 string 以其它任何值开头，则基数是10 (十进制)。

另外，radix的范围是2-36，其中传入0和10以及不传该参数的效果是相同的，都是转换为十进制。有人问传入1会怎样，得到的永远是NaN

|| 与 &&

编程语言中一般都存在逻辑运算符||与&&。但在JS中，或许你需要重新认识一下它们俩。

我们一直都认为在JS中，||与&&返回的是布尔值。但是在上一节中，你应该知道if(..)语句会对非布尔值进行隐式强转，所以说它们俩返回的是布尔值就应该受到质疑了。而上面也提到||和&&左边的操作数会进行ToBoolean抽象操作，因此它们的真面目可以说是被揭穿了。它们俩返回的并不是布尔值，而是两个操作数中的一个值（且仅一个）。我们之所以认为它们返回的是布尔值，是因为它俩一般用在if(..)语句中，而if(..)最后会对它们俩返回的值进行隐式强转以致我们误会了它俩。

  • a || b，如果a不是布尔值，那么||会对a（第一个操作数）进行ToBoolean抽象操作得到布尔值

    • 如果a或a强转的布尔值为true，那么就返回a（第一个操作数的值）
    • 否则，返回b（第二个操作数的值）

  • a && b，如果a不是布尔值，那么&&会对a（第一个操作数）进行ToBoolean抽象操作得到布尔值

    • 如果a或a强转的布尔值为true，那么就返回b（第二个操作数的值）
    • 否则，返回a（第一个操作数的值）

所以在JS中我们可以认为，||与&&是一个选择器。

宽松相等与严格相等

相等比较的正解

在我看来，相等比较是JS中比较有趣的一个地方，它居然有===这种新技能。刚认识==(loose equals)与===(strict equals)时，和很多人一样，认为==是检查值是否相等，而===是检查值以及类型是否相等。其实，这是有偏差的，正确来说：==允许在相等比较中进行强制类型转换，而===不允许。有了上面积累的内功，是不是发现这样才是正解。另外，==与===都会检查操作数的类型，区别在于对类型不同时处理情况不同。

抽象相等比较算法

ECMA规范中对x==y的行为给出了一个算法，叫抽象相等比较算法。

如果Type(x)与Type(y)相同，那么就直接比较它们是否相等。不过需要注意的是

  • NaN == NaN // false （因为NaN是唯一一个非自反的值）
  • +0 == -0 // true
  • 两个对象指向同一个值（引用同一个值）才视为相等

抽象相等比较算法需要重点了解的在于Type(x)与Type(y)不相同的情况，这时候就会发生隐式强制类型转换，会将其中一个值或两个值强转为同一类型再进行比较。

以下的情况都是x与y类型不相同的情况

字符串与数字

上面是10转为”10”，还是”10”转为10呢？规范中指出

If Type(x) is Number and Type(y) is String, return the result of the comparison x == ToNumber(y).

If Type(x) is String and Type(y) is Number, return the result of the comparison ToNumber(x) == y.

  • 如果Type(x)是数字，Type(y)是字符串，返回 x == ToNumber(y) 的结果
  • 如果Type(x)是字符串，Type(y)是数字，返回 ToNumber(x) == y 的结果

布尔类型与其他类型

If Type(x) is Boolean, return the result of the comparison ToNumber(x) == y.

If Type(y) is Boolean, return the result of the comparison x == ToNumber(y).

  • 如果Type(x)是布尔类型，返回 ToNumber(x) == y 的结果
  • 如果Type(y)是布尔类型，返回 x == ToNumber(y) 的结果

根据上面的规范我们应该可以知道为什么会得到false。

a == b，因为b是布尔类型，所以转化为a == ToNumber(b)

因为ToNumber(b)=1，所以转化为 a == 1

因为变量a是字符串，而1是数字，所以转化为ToNumber(a) == 1

最终转化为 10 == 1，结果为false就很清楚了。同样的道理"10" == false返回的值也是false

对象与非对象

Object Type 与 non-Object Type

If Type(x) is either String or Number and Type(y) is Object, return the result of the comparison x == ToPrimitive(y).

If Type(x) is Object and Type(y) is either String or Number, return the result of the comparison ToPrimitive(x) == y.

  • 如果Type(x)是字符串或数字，并且Type(y)是对象，则返回 x == ToPrimitive(y) 的结果
  • 如果Type(x)是对象，并且Type(y)是字符串或数字，则返回 ToPrimitive(x) == y 的结果

这里之所以只提到字符串和数字，是因为前面我们介绍过布尔类型会先进行通过ToNumber强转为数字。

更全面的JS相等比较

小小结

  • 如果两边的值存在true或false，千万不要使用==
  • 如果两边的值存在[]、""、0，尽量不要使用==

上面两种情况最好用===来避免我们意料之外的强制类型转换。

这节最后插播一个tip：switch(..)语句中的case相等比较并不会发生隐式强制类型转换，即等价于===

抽象关系比较

最后我们来深入了解一下关系比较 e.g x < y

  • (1) 对a与b进行ToPrimitive操作，由于valueOf()不能返回简单基本类型，所以调用toString()。a与b调用toString()返回的都是"[object Object]"。因此，根据字母顺序比较，a<b和a>b都不成立
  • (2) 既然a、b的toString()返回的字符串值相同，为什么不相等？我们回想一下就会可以明白，如果a、b都是对象，要令a == b相等，只有a与b令指向同一个对象才成立。
  • (3) 既然 a < b 和 a == b都是false，为什么a <= b 和 a >= b是true呢？因为规范指出：

    • Let r be the result of performing abstract relational comparison rval < lval with LeftFirst equal to false. If r is true or undefined, return false. Otherwise, return true.

    • leftval<=rightval将会处理为rightval<leftval，如果rightval<leftval结果为true或者undefined就返回false，即如果为false就返回true（即将结果取反）
    • 因此根据规范，a<=b将会处理为!(b<a)，其中b<a为false，取反后得到结果为true，所以a<=b为true。
    • 我们对a<=b的认识应该从“小于等于”的转变为“不大于”，即从!(a>b)转变为!(b<a)。对待a>=b同理。

小结

其实无论强制类型转换是显式还是隐式，它们都是根据规范的规则调用抽象操作。显式强制类型转换明确地告诉我们发生了什么类型转换，提高代码可读性和可维护性，隐式强制类型转换，虽然是其他操作的副作用，但是一定程度上它也有助于提高代码可读性。
